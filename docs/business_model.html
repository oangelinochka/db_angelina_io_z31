Модель бізнес-об’єктів

Модель бізнес-об’єктів — це концептуальне представлення основних логічних одиниць (об'єктів), які використовуються в інформаційній системі для управління проєктами. Кожен об’єкт має чітко визначену роль, набір атрибутів та взаємозв’язки з іншими об’єктами системи. Ця модель є фундаментом для побудови бази даних, розробки інтерфейсів користувача та бізнес-логіки системи.

Основні бізнес-об’єкти системи:

1. Користувач (User)

Користувач — це будь-яка особа, яка має доступ до системи. Система підтримує кілька типів користувачів: адміністратор, менеджер проєкту, виконавець (розробник, дизайнер, тестувальник тощо). Кожна роль має свій набір функціональних можливостей.

  • Атрибути: user_id, ім’я, email, пароль (зашифрований), роль, дата реєстрації, статус активності.
  • Приклад: Іван Коваленко, email: ivan.kovalenko@pmtool.com, роль: менеджер.
2. Проєкт (Project)

Проєкт — це сукупність задач, об’єднаних спільною метою. Кожен проєкт створюється менеджером і містить опис, цілі, терміни виконання, статус та список виконавців. Це ключовий об’єкт у системі.

  • Атрибути: project_id, назва, опис, дата створення, дедлайн, статус (активний, завершено, в очікуванні), власник (менеджер).
  • Приклад: “Сайт для стартапу X” – дедлайн: 20.08.2025, статус: активний.
3. Задача (Task)

Задача — це конкретна дія або підзадача, яка виконується в межах одного проєкту. Задачі призначаються конкретним виконавцям, мають свій пріоритет, термін, статус та опис. Можуть бути згруповані в етапи або спринти.

  • Атрибути: task_id, заголовок, опис, проєкт_id, виконавець_id, пріоритет, дедлайн, статус (нове, в роботі, виконано, відкладено), дата створення, останнє оновлення.
  • Приклад: “Розробити головну сторінку” – пріоритет: високий, виконавець: Юрій Бондаренко.
4. Коментар (Comment)

Коментар — це повідомлення, яке залишають користувачі для обговорення задач або уточнення деталей. Це важливий інструмент для командної комунікації.

  • Атрибути: comment_id, автор_id, текст, дата, задача_id.
  • Приклад: “Я оновив логіку API — перевірте, будь ласка.”
5. Файл (Attachment)

Вкладення — це будь-який тип файлів, які прикріплюються до задач або коментарів: макети, документи, зображення, звіти.

  • Атрибути: file_id, ім’я файлу, тип, шлях до файлу, задача_id або коментар_id.
  • Приклад: “design_v2.png” – прикріплений до задачі #14.
6. Повідомлення (Notification)

Повідомлення — необов’язковий, але корисний об’єкт для реалізації системи сповіщень користувача про події: нова задача, змінено статус, додано коментар тощо.

  • Атрибути: notification_id, тип, користувач_id, дата, статус прочитання.

Зв’язки між бізнес-об’єктами:

  • Один користувач може створювати кілька проєктів.
  • Один проєкт має багато задач.
  • Кожна задача призначається одному користувачу, але один користувач може мати багато задач.
  • Кожна задача може мати багато коментарів і файлів.
  • Кожен користувач може залишати багато коментарів.

Висновок

Побудова чіткої моделі бізнес-об’єктів дозволяє краще структурувати систему, визначити логіку зберігання та обробки даних, спростити розробку інтерфейсів та забезпечити масштабованість. Завдяки цій моделі, реалізація REST API, SQL-запитів та контролю ролей стає більш прозорою та ефективною.
